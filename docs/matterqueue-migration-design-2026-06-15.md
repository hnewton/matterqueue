# Matterqueue Migration Pass Design
*Version 1.0 — 2026-06-15*
*Status: Locked — Stage 1 reference*

---

## Overview

The migration tool converts the existing aggregate capsule document and idea index into individual MD files conforming to the Matterqueue front matter schema. It is a standalone HTML file using the File System Access API — same stack as the viewer, no install, no terminal.

Two passes:
- **Pass 1** — generates all MDs from source documents. Presents suggestions for human confirmation where judgment is required. Writes confirmed MDs to the output folder.
- **Pass 2** — audits the generated corpus. Produces a `_review.md` gap report. Does not generate new MDs.

---

## Source Documents

| Source | Role |
|---|---|
| `idea_capsule_aggregate.md` | Primary content source. Contains full Idea Capsule content per idea. |
| `idea_capsule_index.md` | State and classification source. Contains doc_level signals, bucket verdicts, absorption status, type flags. |

Both documents are loaded via File System Access API at migration tool open. The tool parses them in memory — neither is modified.

## Field Mapping — Index to Front Matter

The index provides classification signals. The tool reads these and maps to front matter fields.

| Index signal | Front matter field | Mapping rule |
|---|---|---|
| Idea name | `name` | Direct. Exact string from index. |
| ⚙️ flag | `type` | `engine` |
| 🔧 flag | `type` | `workflow` |
| No flag | `type` | `concept` (default) |
| Full Capsule (★ or standard) | `doc_level` | `capsule` |
| Mini Capsule | `doc_level` | `spark` |
| Infrastructure Note | `doc_level` | `spark` |
| Advanced Capsule (★★) | `doc_level` | `brief` |
| Absorbed status in index | `state` | `absorbed` |
| Dissolved status in index | `state` | `retired` |
| Parked / Shelf It verdict | `state` | `parked` |
| Active / Dig In verdict | `state` | `active` |
| absorbed_by reference in index | `absorbed_by` | Direct text from index absorption note |
| Today's date | `created_date` | YYYY-MM-DD — default for all migrated MDs |
| Today's date | `modified_date` | YYYY-MM-DD — updated on every subsequent write-back |

## Field Mapping — Aggregate to Idea Capsule Body

The aggregate provides capsule content. The tool maps aggregate fields directly to Idea Capsule body sections.

| Aggregate field | Idea Capsule field | Mapping rule |
|---|---|---|
| Name | `Name` | Direct |
| One Phrase | `One Phrase` | Direct |
| Core Sentence | `Core Sentence` | Direct |
| Hook | Folded into `One Paragraph` generation (see below) | Not a standalone field in new schema |
| Key Insights | `Key Insights` | Direct — preserve as bullet list |
| User | `User` | Direct |
| Model | `Model` | Direct |
| Why It's Real | `Why It's Real` | Direct — preserve as bullet list |
| Flags | `Flags` | Direct |
| Synergy (freeform text) | `Synergy` body section | Direct — preserve as written; also used as signal for synergies: suggestion (see below) |
| Yes-and | Dropped | Triage artifact — not migrated |
| Recommendation | Dropped | Replaced by state + bucket flags |
| Next Step | Seed for actions: suggestion (see below) | Not migrated directly |

## Generated Fields — System Drafts, Human Confirms

### One Paragraph

Generated by the tool from existing capsule content. Not in the aggregate schema — new addition.

**Generation prompt sent to generation layer:**
> Using the following fields from this idea's capsule — Core Sentence, Hook (if present), Key Insights, Why It's Real — write a One Paragraph external pitch. Open with the sharpest entry point that makes someone lean in. Expand into who does what they couldn't before. Close with why it's real or why now. Three to five sentences. Written to be spoken or sent.

Output is presented to the human in the confirmation UI before writing to the MD. Human can edit inline or accept as-is.

If generation fails or produces poor output: inline placeholder inserted — `[FILL: one paragraph pitch — open with hook, expand core sentence, close with why now]`

---

### uses: and synergies: suggestions

**Source signal:** The aggregate Synergy field contains freeform relational text per idea. The tool also cross-references all idea names across the full corpus.

**uses: suggestion logic:**
- Scan idea body for explicit engine or workflow name mentions
- Scan Synergy field for phrases indicating dependency ("powered by", "uses", "relies on", "built on")
- Cross-reference against known engine and workflow names in the index
- Present candidates as a checklist: `[ ] InsightMutant — mentioned in Synergy section`

**synergies: suggestion logic:**
- Extract named idea references from the Synergy field
- Scan Key Insights and Why It's Real for idea name mentions
- Cross-reference against full idea name list
- Present candidates as a checklist: `[ ] EF Coach — referenced in Synergy section`

**Confirmation UI:**
Each MD's suggestions are presented as a grouped checklist before writing. Human checks to confirm, unchecks to reject, and can type additional names not suggested. Confirmed values written to front matter as exact name strings.

**Bidirectional note:** When human confirms a synergy relationship A → B, the tool flags that B's synergies: list should also include A. This is surfaced in Pass 2 gap report rather than auto-written — B may not have been reviewed yet.

---

### actions: suggestions

**Source signal:** Next Step field from aggregate (freeform text). Prototype signals from Flags or Why It's Real sections. Research signals from Key Insights or open questions in aggregate.

**Suggestion logic:**
- Parse Next Step as one or more candidate action items
- Infer type: if Next Step contains "prototype", "build", "test" → `prototype`; "research", "validate", "explore", "map" → `research`; "decide", "determine", "choose" → `decision`; default → `task`
- Set state: `open` for all suggested actions
- Set position: sequential integers starting at 1 — human reorders in Pass 2

**Confirmation UI:**
Suggested actions presented as an editable list before writing. Human can accept, edit description, change type, delete, or add new items. Confirmed actions written to front matter YAML.

If no Next Step in aggregate: actions: field omitted from front matter for that MD. Human adds in Pass 2 if needed.

---

### focus: / try_it: / map_it: suggestions

**focus:** — suggested as `true` only for ideas currently marked active with a Dig In verdict. Human confirms.

**try_it:** — suggested as `true` if any confirmed action has `type: prototype`, or if aggregate contains prototype signal language. Human confirms.

**map_it:** — suggested as `true` if any confirmed action has `type: research`, or if aggregate contains mapping/research signal language. Human confirms.

All three presented as checkboxes in the confirmation UI alongside action suggestions.

---

### Brief update on absorption (absorbing idea)

When an absorbed idea is processed, the tool drafts a Brief entry for the absorbing idea:

**Draft template:**
> **Brief — [migration date]**
>
> [Absorbed idea name] absorbed into this concept. [Absorbed idea's Core Sentence]. Key elements carried forward: [Key Insights from absorbed idea — top 2]. See [absorbed-filename.md] for full history.

Presented to human for confirmation or edit before writing. If human declines: not written; flagged in `_review.md` for manual authoring.

## MD Generation Rules by Doc Level

### spark

Minimal MD. Front matter only + one sentence body.

```yaml
---
name: [name]
type: [concept / engine / workflow]
doc_level: spark
state: [active / parked / absorbed / retired]
absorbed_by: [if absorbed]
created_date: YYYY-MM-DD
modified_date: YYYY-MM-DD
---
```

Body:
```markdown
[One sentence from aggregate or index. If none available: [FILL: one sentence description]]
```

No Idea Capsule section. No logs (empty sections created with headers only).

---

### capsule

Full front matter. Full Idea Capsule body section. Empty log sections.

```yaml
---
name: [name]
type: [concept / engine / workflow]
doc_level: capsule
state: [state]
absorbed_by: [if absorbed]
focus: [true / false]
try_it: [true / false]
map_it: [true / false]
uses:
  - [confirmed names]
synergies:
  - [confirmed names]
todoist: false
created_date: YYYY-MM-DD
modified_date: YYYY-MM-DD
actions:
  - description: [confirmed action]
    type: [type]
    state: open
    position: [n]
---
```

Body:
```markdown
## Idea Capsule — [migration date]

**Name** — [name]
**One Phrase** — [from aggregate]
**Core Sentence** — [from aggregate]
**One Paragraph** — [generated or FILL placeholder]

**Key Insights**
[from aggregate]

**User** — [from aggregate]

**Model** — [from aggregate]

**Why It's Real**
[from aggregate]

**Flags** — [from aggregate]

**Synergy**
[from aggregate — freeform text preserved as written]

---

## Action Log

---

## Decision Log

---

## Open Questions Log
```

---

### brief

Same as capsule MD generation plus a placeholder Brief section below the Idea Capsule.

```markdown
## Brief — [FILL: date when brief content is added]

[FILL: working notes, session outputs, signals. Add above this section for newer entries.]
```

Note: Brief content is not in the aggregate — this section is always a placeholder at migration. Human populates in Pass 2 or in subsequent sessions.

---

### absorbed

Same as capsule or spark generation (based on what existed in aggregate), with:
- `state: absorbed`
- `absorbed_by: [name]` populated
- Note appended at top of body:

```markdown
> **Absorbed** — This concept was absorbed into [absorbing idea name] on [migration date].
> See [absorbing-idea-filename.md] for current development.
```

All prior content preserved unchanged below the note.

## Confirmation UI — Flow

The migration tool uses batch confirmation mode. All ideas are processed and pre-filled with suggestions before the human reviews. The human scans and corrects outliers rather than confirming each item individually.

**Batch flow:**

1. **Parse** — tool reads aggregate and index, processes all ideas, generates suggestions for every field.
2. **Pre-fill** — all suggestions are pre-confirmed by default. The review UI shows every idea in a scrollable list with its generated front matter and body fields populated.
3. **Scan and correct** — human scrolls through, editing only what needs changing. Every field is editable inline. No mandatory stops.
4. **Confirm all** — single "Confirm All & Generate" button at top and bottom of the list. Writes all MDs to output folder in one pass.
5. **Skip individual ideas** — human can uncheck any idea to exclude it from the current run. Unchecked ideas are flagged in `_review.md`.

**Inline edit controls per idea:**
- Dropdown for `type`, `doc_level`, `state`
- Text input for `absorbed_by` (visible only when state: absorbed)
- Tag inputs for `uses:` and `synergies:` — pre-filled suggestions shown as removable tags; type to add more
- Editable text area for One Paragraph — generated draft shown; human edits inline
- Action list — each suggested action shown with editable description, type dropdown, delete button; "Add action" button at bottom
- Checkboxes for `focus:`, `try_it:`, `map_it:`

**Progress indicator:** `[n] of [total] reviewed / [n] skipped / Ready to generate`

**Batch confirm safety:** Before writing, tool displays summary: "Generate [n] MDs to [folder path]? [n] skipped." Confirm / Cancel.

## Pass 2 — Corpus Audit

Runs after Pass 1 is complete. Reads all generated MDs. Produces `_review.md`.

### Audit checks

**Missing One Paragraphs**
Flag every MD where One Paragraph contains `[FILL:` placeholder.
```
❌ thinkmutant.md — One Paragraph not confirmed
```

**Unconfirmed placeholders**
Scan all body content for `[FILL:` strings. List by file and field.
```
❌ shapesake.md — Brief section not populated
❌ walkframe.md — body one-sentence description missing
```

**Unconfirmed actions**
Flag MDs where actions: is absent but Next Step was present in aggregate — may indicate actions were skipped rather than intentionally omitted.
```
⚠️ cobaltbat.md — Next Step present in aggregate but no actions in front matter. Intentional?
```

**Synergy bidirectionality gaps**
For every confirmed synergy A → B, check that B's synergies: list includes A. Flag mismatches.
```
⚠️ patterntopia.md lists thinkmutant in synergies: but thinkmutant.md does not list patterntopia. Add?
```

**uses: name resolution**
For every name in uses: across all MDs, verify a matching MD exists with that exact name. Flag unresolved references.
```
❌ siteframe.md uses: "Semantic matching engine" — no MD found with that name. Create MD or correct name?
```

**synergies: name resolution**
Same check for synergies: lists.

**Absorbed ideas without Decision Log entry in absorbing MD**
For every absorbed MD, check that the absorbing MD's Decision Log contains an entry referencing the absorbed idea.
```
⚠️ manhole.md absorbed into shapesake — shapesake.md Decision Log has no absorption entry. Add?
```

**Absorbed MDs missing absorbed_by**
Flag any MD with `state: absorbed` but no `absorbed_by` value.
```
❌ syllably.md — state: absorbed but absorbed_by is empty
```

**Name uniqueness check**
Scan all `name` fields across all MDs. Flag any duplicates.
```
❌ Duplicate name detected: "EF Coach" appears in ef-coach.md and ef-coach-pro.md
```

**focus: flag count**
Count MDs with `focus: true`. Flag if more than 2.
```
⚠️ 3 concepts marked focus: true — patterntopia, thinkmutant, the-current. Recommended limit is 2.
```

## _review.md Format

Generated at folder root after Pass 2. Three tiers — blocking first, then non-blocking, then suggested improvements. Structured for human triage.

```markdown
# _review.md — Matterqueue Migration Gap Report
*Generated: YYYY-MM-DD*
*Pass 2 audit of [n] MDs*

---

## ❌ Blocking — fix before viewer build

Items that will cause viewer errors, broken references, or silent data loss.
Fix these before opening the viewer.

- thinkmutant.md — uses: "Semantic matching engine" resolves to no MD
- syllably.md — state: absorbed but absorbed_by is empty
- [duplicate name conflict if any]

---

## ⚠️ Non-blocking — review before first use

Items that are incomplete but won't break the viewer. Gaps in content
or coverage that will surface during normal use.

- cobaltbat.md — Next Step present in aggregate but no actions in front matter
- patterntopia.md — One Paragraph is [FILL:] placeholder
- shapesake.md — Brief section not populated
- thinkmutant.md lists patterntopia in synergies: but patterntopia.md does not list thinkmutant

---

## 💡 Suggested — optional improvements

Items that are technically complete but could be richer.

- walkframe.md — spark level; consider expanding to capsule when time permits
- 3 concepts marked focus: true — patterntopia, thinkmutant, the-current; recommended limit is 2

---

## ✅ Confirmed clean

[n] MDs passed all checks with no gaps.

---

*Delete this file when all ❌ and ⚠️ items are resolved.*
```

## Migration Tool — Technical Notes

**Stack:** Single HTML file. Vanilla JavaScript. File System Access API. No dependencies, no npm, no server.

**File System Access API usage:**
- `showOpenFilePicker()` — select aggregate and index source files
- `showDirectoryPicker()` — select output folder (matterqueue/venture-portfolio/)
- `FileSystemDirectoryHandle.getFileHandle(name, { create: true })` — create each MD
- `FileSystemWritableFileStream` — write MD content

**YAML generation:** Handwritten string templates. No YAML library needed — front matter is simple enough that template strings produce valid YAML without a parser. Validate output visually on first run.

**Markdown parsing for source documents:** Split on heading patterns (`## [Idea Name]`) to isolate per-idea content blocks. Field extraction via line-prefix matching (`**Core Sentence** —`). Robust to minor formatting variation in aggregate.

**One Paragraph generation:** Calls the same generation approach used in the viewer (in-browser, no server). If unavailable, falls back to `[FILL:]` placeholder.

**Error handling:** Any MD that fails to write is logged in the console and flagged in `_review.md`. Migration continues — no full stop on single file error.

**Re-runnability:** The tool can be re-run. It checks for existing MDs in the output folder and prompts: "Overwrite existing MDs? (Yes / Skip existing / Cancel)". Skip existing is the safe default for re-runs after partial completion.

## Pre-Migration Checklist

Before running the migration tool:

- [ ] Schema spec reviewed and locked (`matterqueue-schema-2026-06-15.md` v1.3)
- [ ] Viewer built and verified against hand-crafted test MDs — schema gaps resolved before migration runs
- [ ] `_system.md` created in output folder with correct domain config
- [ ] `matterqueue.md` created manually in output folder
- [ ] `compilation-matterqueue.md` created manually in output folder
- [ ] `build-matterqueue.md` created manually in output folder
- [ ] Aggregate document is the most recent version
- [ ] Index document is the most recent version
- [ ] Output folder exists with manually created files in place
- [ ] Chrome is the active browser (File System Access API required)
